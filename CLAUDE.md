## What Is This?

Dollarbucks is a parent-child allowance and chore management PWA. Parents define chores for each child; chores either pay a **fixed amount** (ledger entry posted immediately on toggle) or contribute **weighted credit toward a weekly allowance** (distributed during weekly reset). The app is mobile-first (375×667px target) with offline support via service worker.

## Commands

```bash
bun dev          # Start dev server (localhost:5173)
bun run build    # tsc -b && vite build (type-check + bundle)
bun run lint     # ESLint
bun test:unit    # Vitest (unit tests)
bun test:e2e     # Playwright (E2E, auto-starts dev server)
bun test:e2e:ui  # Interactive Playwright runner
```

**Run a single unit test:**

```bash
bun test:unit -- src/features/chores/useChoreActions.test.ts
```

**Run a single E2E suite:**

```bash
bun test:e2e -- chores.spec.ts
```

## Architecture

### Feature-First Structure

All code lives under `src/features/<domain>/`, keeping types, store, business logic, and components co-located:

- `features/app/` — global date state (`currentDate`, `currentWeekStartDate`)
- `features/children/` — child profiles (name, avatar, weekly allowance amount)
- `features/chores/` — chore definitions, completion tracking, toggle logic
- `features/ledger/` — financial transaction log (add/remove entries, balance calc)

`src/components/ui/` holds shared primitives (Button, Input, Modal, ToggleGroup). `src/pages/` are thin route targets that compose feature components.

### State Management (Zustand + localStorage)

Four persisted stores, each with its own localStorage key:

| Store               | Key                    | Notes                                        |
| ------------------- | ---------------------- | -------------------------------------------- |
| `useChildrenStore`  | `dollarbucks-children` |                                              |
| `useChoresStore`    | `dollarbucks-chores`   | v1 with migration from legacy `isComplete`   |
| `useLedgerStore`    | `dollarbucks-ledger`   |                                              |
| `useAppStore`       | `dollarbucks-app`      | Only `currentWeekStartDate` is persisted     |

### Two Earning Models (Critical Distinction)

Chores have an `earningType` field that drives fundamentally different payment flows:

- **`fixed`**: `toggleChore()` immediately posts/reverses a ledger entry when marking complete/incomplete.
- **`allowance`**: Completion is recorded but no ledger entry is created until `useWeeklyReset` runs, which distributes the child's `weeklyAllowance` weighted by completion percentage.

Any code touching chore completion **must branch on `earningType`**.

### Key Pure Functions

Business logic is intentionally decoupled from React for testability:

- `toggleChore(chore, date?)` — `features/chores/useChoreActions.ts` — main action for marking chores done
- `isChoreComplete(chore, date)` — `features/chores/completionHelpers.ts`
- `getCompletionKey(chore, date)` — returns ISO date for daily chores, `"week"` for weekly chores
- `getToday()` — `features/chores/dateHelpers.ts` — returns overridden date in dev, real date in prod

### Date Handling

All date calculations use **local time** (not UTC) to avoid timezone midnight drift. `getMondayOfThisWeek()` in `dateHelpers.ts` is the canonical week-start calculator. The `DevDatePicker` component (lazy-loaded, dev-only) lets you override the current date to test weekly reset flows.

## Testing

- **Unit tests** (Vitest + jsdom): `*.test.ts` alongside the source in `features/`. Test pure functions and store logic.
- **E2E tests** (Playwright): `e2e/*.spec.ts`, mobile Chromium at 375×667px with touch enabled. 9 suites covering the main user flows.
- Vitest is configured with `globals: true` and excludes the `e2e/` directory.

**When to run which:**

- `test:unit` — run constantly while writing logic or components
- `test:e2e` — run before committing or when something looks wrong in the browser
- `test:e2e:ui` — use when debugging a flaky or failing E2E test

## PWA / Build Notes

- Service worker is generated by `vite-plugin-pwa` (workbox). Caches fonts, assets, and runtime requests.
- The PWA manifest and service worker are only present in production builds — `bun dev` skips them.
- Tailwind brand tokens (navy, blue, orange, yellow, green, cream) are defined in `tailwind.config.js` theme extensions — use these rather than arbitrary color values.
